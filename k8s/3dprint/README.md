# 3dprint Kubernetes Deployment

This directory provides a minimal Kubernetes deployment for FilaOps in the
`3dprint` namespace, including:

- PostgreSQL stateful storage
- Backend and frontend deployments
- Notification microservice deployed from source via ConfigMap
- Init-time backend migrations
- Splunk OpenTelemetry Python auto-instrumentation on the backend
- OpenTelemetry Operator Node.js auto-instrumentation on the frontend service
- Same-origin browser trace export from the frontend to the cluster collector
- Optional Playwright load-smoke runner job

## Quick start (turnkey deploy)

The fastest path from a bare Kubernetes cluster to a working demo with all
integrations (OTel collector, app, ThousandEyes, Cilium/Hubble, Splunk
dashboards) is the single `deploy` command:

```sh
cp scripts/demo/.env.example scripts/demo/.env.local
# Edit .env.local: fill in SPLUNK_REALM, SPLUNK_ACCESS_TOKEN,
#   THOUSANDEYES_BEARER_TOKEN, FRONTEND_URL, DEMO_ADMIN_EMAIL, DEMO_ADMIN_PASSWORD

bash scripts/demo/3dprint-observability.sh deploy
```

This will:

1. Install cert-manager, OpenTelemetry Operator, and Splunk OTel Collector via Helm
2. Create secrets and ConfigMaps, apply kustomize, wait for pods
3. Provision ThousandEyes tests with metric and trace streams to Splunk
4. Auto-detect Cilium and configure Hubble metrics scraping
5. Create Splunk Observability dashboards with trace correlation panels
6. Print all dashboard and handoff links

To tear everything down:

```sh
bash scripts/demo/3dprint-observability.sh teardown
# Add TEARDOWN_OTEL=true to also remove the OTel collector and operator
```

## Advanced / Manual Setup

If you prefer to set up each component individually instead of using `deploy`:

### Build and push images

Log in to ECR, then build and push all three images:

```sh
aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin 637423309390.dkr.ecr.us-east-1.amazonaws.com
docker build --platform linux/amd64 -t 637423309390.dkr.ecr.us-east-1.amazonaws.com/filaops-backend:env-3dprint-20260402-asset-order-ingest-fix-d9b6388 ./backend
docker push 637423309390.dkr.ecr.us-east-1.amazonaws.com/filaops-backend:env-3dprint-20260402-asset-order-ingest-fix-d9b6388
docker build --platform linux/amd64 -t 637423309390.dkr.ecr.us-east-1.amazonaws.com/filaops-frontend:demo-20260401-rum-link-d9b6388 ./frontend
docker push 637423309390.dkr.ecr.us-east-1.amazonaws.com/filaops-frontend:demo-20260401-rum-link-d9b6388
docker build --platform linux/amd64 -t 637423309390.dkr.ecr.us-east-1.amazonaws.com/filaops-playwright:load-smoke-amd64-20260401-d9b6388 -f frontend/Dockerfile.playwright ./frontend
docker push 637423309390.dkr.ecr.us-east-1.amazonaws.com/filaops-playwright:load-smoke-amd64-20260401-d9b6388
```

### Create secrets

For a demo environment, use the helper script:

```sh
bash k8s/3dprint/create-demo-secret.sh
```

By default it uses:

- `DB_PASSWORD=C1sco12345`
- a random `SECRET_KEY`
- `DEMO_ADMIN_EMAIL=admin@example.com`
- `DEMO_ADMIN_PASSWORD=C1sco12345`
- `NOTIFICATION_SERVICE_TOKEN=demo-notification-token`

This helper is the source of truth for secrets. The kustomization intentionally
does not include `secret.yaml`, so `kubectl apply -k` will not overwrite the
generated values.

### Publish the notification service bundle

The notification service is deployed from the checked-out source tree, so
publish its files into a ConfigMap before applying:

```sh
bash k8s/3dprint/create-notification-service-configmap.sh
```

### Apply

```sh
kubectl apply -k k8s/3dprint
```

### Access the demo

Port-forward the frontend service locally:

```sh
kubectl -n 3dprint port-forward svc/frontend 8080:80
```

## Run Playwright load smoke in-cluster

1. Publish the current benchmark manifest into a ConfigMap:

```sh
bash k8s/3dprint/create-loadgen-manifest-configmap.sh
```

By default that uses `scripts/loadgen/manifest.3dprint.json`. Override
`MANIFEST_PATH` if you want to publish a fresh manifest generated by the
benchmark seeder.

To seed the benchmark dataset in-cluster and refresh `loadgen-manifest`
automatically from the job output, run:

```sh
bash k8s/3dprint/run-seed-loadgen.sh
```

The seeder job:

- uses the currently deployed backend image by default
- prunes older finished `seed-loadgen-manifest-*` jobs before creating a new one
- leaves completed jobs behind for 10 minutes by default via `JOB_TTL_SECONDS=600`
- updates the `loadgen-manifest` ConfigMap after the job completes

To run the API load against the in-cluster services with k6, run:

```sh
bash k8s/3dprint/run-k6-loadgen.sh
```

The k6 job:

- publishes `scripts/loadgen/main.js` into a temporary ConfigMap automatically
- mounts `loadgen-manifest/manifest.json` at `/config/manifest.json`
- defaults to `BASE_URL=http://frontend` and `API_BASE_URL=http://frontend/api/v1`
- defaults to `WORKLOAD=service-map` so the demo path keeps
  `frontend`, `backend`, `order-ingest`, `pricing-service`, `asset-service`,
  and `notification-service` active on the map
- prunes older finished `k6-loadgen-*` jobs before creating a new one
- leaves completed jobs behind for 10 minutes by default via `JOB_TTL_SECONDS=600`

Use `WORKLOAD=all bash k8s/3dprint/run-k6-loadgen.sh` only when you explicitly
want the heavier benchmark profile. The default `service-map` path is meant to
shape the APM map without saturating the single backend pod, and now uses a
short 1-2 second pause plus direct root probes to keep all demo services warm.

2. Launch the smoke job:

```sh
bash k8s/3dprint/run-playwright-smoke.sh
```

The runner:

- pulls `filaops-playwright`
- defaults to the dedicated amd64 smoke tag instead of the shared demo app tag
- targets the in-cluster frontend service at `http://frontend`
- reads admin credentials from `filaops-secrets`
- mounts `loadgen-manifest/manifest.json` at `/config/manifest.json`
- prunes older finished Playwright jobs before creating a new one
- leaves completed jobs behind for 10 minutes by default via `JOB_TTL_SECONDS=600`

To prune finished loadgen, seed, and Playwright jobs on demand:

```sh
bash k8s/3dprint/cleanup-finished-jobs.sh
```

## Run backend integration smoke in-cluster

To verify the live backend together with `asset-service` and `order-ingest`,
run:

```sh
bash k8s/3dprint/run-backend-service-smoke.sh
```

The runner:

- executes inside the live backend pod selected by `BACKEND_TARGET`
- checks `/health` on `backend`, `asset-service`, and `order-ingest`
- logs into the backend with `DEMO_ADMIN_EMAIL` and `DEMO_ADMIN_PASSWORD` from `filaops-secrets`
- uploads, fetches, and deletes a product image through the backend API
- imports a one-line CSV order through the backend API using `ORDER_IMPORT_SKU` when provided, otherwise auto-discovering an active product SKU from the database

Useful overrides:

- `NAMESPACE=...`
- `BACKEND_TARGET=deploy/backend`
- `BACKEND_BASE_URL=...`
- `ASSET_SERVICE_BASE_URL=...`
- `ORDER_INGEST_BASE_URL=...`
- `ORDER_IMPORT_SKU=...`
- `MODE=full|order-import-only`
- `RUN_ID=demo-...`
- `EXPECT_ORDER_IMPORT_STATUS=200|502`

For a static manifest instead of the helper script, apply:

```sh
kubectl delete job -n 3dprint playwright-load-smoke --ignore-not-found
kubectl apply -f k8s/3dprint/playwright-smoke-job.yaml
kubectl logs -n 3dprint -f job/playwright-load-smoke
```

## Cross-observability demo workflow

If you used the turnkey `deploy` command above, all integrations are already
provisioned. Skip straight to the demo sequence.

If you set up manually, provision the observability integrations first:

```sh
cp scripts/demo/.env.example scripts/demo/.env.local
# fill in credentials
bash scripts/demo/3dprint-observability.sh setup
```

Then run the demo sequence:

```sh
bash scripts/demo/3dprint-observability.sh baseline
bash scripts/demo/3dprint-observability.sh break
bash scripts/demo/3dprint-observability.sh restore
```

Print the current dashboard and handoff links at any point with:

```sh
bash scripts/demo/3dprint-observability.sh links
```

The wrapper provisions:

- the official Cisco ThousandEyes Splunk Observability dashboard package from `scripts/demo/assets/splunk-o11y/thousandeyes-dashboard-package.json`
- `FilaOps Demo - App + ThousandEyes`
- `FilaOps Demo - Policy Impact`
- an optional repo-managed ThousandEyes enterprise agent deployment in-cluster when `THOUSANDEYES_ENTERPRISE_AGENT_ACCOUNT_TOKEN` is set
- a ThousandEyes HTTP test for the external frontend
- a ThousandEyes browser transaction for admin login and order-import navigation
- a ThousandEyes API test for the authenticated admin API path
- a ThousandEyes OpenTelemetry stream that forwards those metrics to Splunk Observability
- a reversible policy from `k8s/3dprint/policies/order-ingest-deny.cnp.yaml` or the broader fallback `k8s/3dprint/policies/order-ingest-deny.netpol.yaml`

Required environment:

- `SPLUNK_O11Y_REALM`
- `SPLUNK_O11Y_API_TOKEN`
- `SPLUNK_O11Y_INGEST_TOKEN` when your org separates dashboard/API and ingest tokens
- `THOUSANDEYES_API_TOKEN`
- `THOUSANDEYES_ACCOUNT_GROUP_ID` if you need to override the repo default of `2114135` (`Splunk Observability Demo`)
- `THOUSANDEYES_AGENT_IDS` or the role-specific overrides below
- `FRONTEND_URL`
- `DEMO_ADMIN_EMAIL`
- `DEMO_ADMIN_PASSWORD`

Useful ThousandEyes overrides:

- `THOUSANDEYES_BROWSER_AGENT_IDS` to keep browser transactions on BrowserBot-capable agents
- `THOUSANDEYES_HTTP_AGENT_IDS`
- `THOUSANDEYES_API_AGENT_IDS`
- `THOUSANDEYES_EXISTING_HTTP_SERVER_TEST_ID`
- `THOUSANDEYES_EXISTING_WEB_TRANSACTION_TEST_ID`
- `THOUSANDEYES_EXISTING_API_TEST_ID`
- `THOUSANDEYES_PIN_EXISTING_TRANSACTION_TESTS=true` when the account group is quota-locked and the existing browser/API tests must be referenced read-only instead of modified
- `THOUSANDEYES_ENTERPRISE_AGENT_ACCOUNT_TOKEN` to let the repo install a second Enterprise Agent into the cluster
- `THOUSANDEYES_ENTERPRISE_AGENT_NAMESPACE`
- `THOUSANDEYES_ENTERPRISE_AGENT_NAME`
- `THOUSANDEYES_ENTERPRISE_AGENT_STORAGE_CLASS`
- `THOUSANDEYES_ENTERPRISE_AGENT_STORAGE_SIZE`

Quota note:

- When `THOUSANDEYES_PIN_EXISTING_TRANSACTION_TESTS=true`, the repo keeps the shared ThousandEyes browser/API test IDs pinned read-only and the custom Splunk dashboards rely on the live FilaOps HTTP synthetic plus backend telemetry for guaranteed data.
- If those pinned browser/API IDs do not actually target the FilaOps demo URLs, the setup skips the imported generic ThousandEyes dashboard package so the demo does not show empty or misleading panels from unrelated tests.

To install or inspect the repo-managed enterprise agent explicitly:

```sh
bash scripts/demo/3dprint-observability.sh agent-install
bash scripts/demo/3dprint-observability.sh agent-status
```

Optional handoff links:

- `SPLUNK_NETWORK_EXPLORER_URL`
- `THOUSANDEYES_DASHBOARD_URL`
- `ISOVALENT_DASHBOARD_URL`
- If `ISOVALENT_DASHBOARD_URL` is unset, the setup tries to auto-discover the built-in `Cilium by Isovalent` and `Hubble by Isovalent` dashboards in Splunk Observability and uses those links in the demo notes when available.

See [`docs/demo/3dprint-cross-observability.md`](../../docs/demo/3dprint-cross-observability.md) for the full presenter script. It is written for Splunk SEs, Solution Architects, and executives, and explains the ThousandEyes vantage points and chosen tests in plain English.

## Observability notes

- Backend traces are injected by the OpenTelemetry operator through:
  `instrumentation.opentelemetry.io/inject-python: 3dprint/filaops-python`
- Notification service traces are injected by the same Python instrumentation
- Frontend service traces are injected by the OpenTelemetry operator through:
  `instrumentation.opentelemetry.io/inject-nodejs: 3dprint/filaops-nodejs`
- Frontend UX telemetry is sent with Splunk RUM for Browser using
  same-origin OTLP export through `/otel/v1/traces`
- Session replay is enabled for the frontend
- Replay privacy defaults are selective:
  most UI text is visible, typed input values stay masked, and high-sensitivity
  routes/components are explicitly masked
- The RUM token is browser-visible by design, so keep it in Kubernetes secrets
  and treat it as a scoped client token rather than a server secret
- Browser-to-backend correlation still depends on the backend returning
  `Server-Timing: traceparent;desc="..."`. If that header is missing, frontend
  RUM and replay still work, but direct navigation from browser sessions into
  backend traces is incomplete
- Backend logs already emit JSON to stdout; the deployment disables file log
  outputs so log collection can come from the container stream
